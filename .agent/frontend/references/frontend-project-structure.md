# Frontend Project Structure

## 目标

本文件用于约束前端项目中的目录结构和文件落点，减少随意落盘，保证：

1. 目录职责清晰
2. 改动范围最小
3. 内容优先就近放置
4. 不为了单次需求升级成全局抽象

## 目录职责

基于当前约束，前端业务代码默认位于 `src/` 下，常见职责如下：

- `src/pages`
  - 页面级代码
  - 页面私有样式、类型、配置、组件优先就近放置
- `src/components`
  - 稳定复用的跨页面组件
  - 不放只服务单页的强业务组件
- `src/api`
  - 业务接口封装
  - 页面不直接散落请求地址
- `src/models`
  - Umi 全局状态
  - 只存真正跨页面共享的数据
- `src/layouts`
  - 布局级代码
- `src/constants`
  - 跨模块共享常量
- `src/utils`
  - 通用工具函数
- `src/types`
  - 跨模块共享类型
- `src/assets`
  - 静态资源
- `src/app.tsx`
  - 应用运行时入口
  - 涉及 Qiankun 生命周期时才允许谨慎修改
- `src/request.tsx`
  - 全局请求配置与拦截逻辑
- `src/.umi`
  - Umi 生成目录，禁止手工写业务代码

## 落点原则

1. 页面私有内容优先放在页面目录内
2. 只有稳定跨页面复用后，才允许进入 `src/components`、`src/utils`、`src/constants`、`src/types`
3. 接口访问统一放进 `src/api`
4. 页面局部状态优先留在页面内，不轻易提升到 `src/models`
5. 类型定义优先靠近使用位置
6. 样式优先跟随页面或组件，不为单次需求创建全局样式目录
7. 若现有文件已能承载改动，优先原地修改，不额外拆目录、拆层级

## 页面目录规范

推荐页面目录结构如下：

```text
src/pages/
  RiskStatement/
    index.tsx
    index.less
    tableConfig.ts
    components/
```

页面目录中适合放：

- `index.tsx`
- 页面私有样式文件
- `components/`
- `constants.ts` / `config.ts`
- `types.ts`
- 页面私有 `hooks.ts`

页面目录中不适合放：

- 跨页面通用组件
- 跨业务通用工具
- 全局状态
- 通用请求封装

## 组件放置规则

放入 `src/pages/<PageName>/components` 的情况：

- 只被一个页面使用
- 与该页面字段、交互、展示强绑定
- 抽出来只是为了降低页面主文件复杂度

放入 `src/components` 的情况：

- 已被多个页面稳定复用
- 职责稳定，不依赖单一页面上下文
- 升级为公共组件后不会引入大量业务特例

明确不建议：

- 只用一次就创建全局组件
- 因为“以后可能复用”就提前抽公共组件
- 默认把页面弹窗、页面表格、页面筛选区提升到 `src/components`

## 状态与类型

状态：

- 筛选项、分页、弹窗开关、临时表单值优先用页面内部状态
- 只有跨页面共享时才考虑进入 `src/models`

类型：

- 页面私有类型放页面目录
- 单个 API 模块专用类型优先贴近 API 文件
- 多页面、多模块共享类型再进入 `src/types`

## 不建议主动新增的目录

除非已证明必要性，否则不要主动新增以下目录：

- `src/services`
- `src/hooks`
- `src/styles`
- `src/lib`
- `src/helpers`
- `src/common`

原因不是这些目录永远不能存在，而是当前体系默认没有稳定使用模式；新增前必须先证明收益，并确保不与现有结构冲突。

## AI 落盘决策顺序

当需要新增文件时，按以下顺序判断：

1. 这个内容是否只服务当前页面
2. 如果是，优先放到对应页面目录
3. 如果不是，它是否已被多个页面稳定复用
4. 若是组件，放 `src/components`
5. 若是工具，放 `src/utils`
6. 若是共享类型，放 `src/types`
7. 若是共享常量，放 `src/constants`

## 明确禁止

- 把业务代码放进 `src/.umi`
- 页面中直接散落请求地址
- 为单次使用创建全局组件
- 把页面私有逻辑默认上升到 `src/utils` 或 `src/components`
- 在没有明确复用价值时新增一层目录或抽象
- 在未形成团队共识前混用 `src/api` 与 `src/services`

## 一句话原则

能就近放就不要放全局，能复用现有目录就不要新开目录，能局部修改就不要为了单次需求升级结构。
