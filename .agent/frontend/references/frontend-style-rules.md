# Frontend Style Rules

## 目标

本文件用于约束前端项目中的样式实现方式，目标是：

1. 保证样式实现与项目现有体系一致
2. 降低全局污染、样式冲突、重复样式和粗暴覆盖带来的维护成本
3. 让 AI 在写页面和组件时优先复用已有能力，而不是临时堆样式

若目标项目已有更高优先级的设计系统、主题体系或样式工程方案，以目标项目自身规则为准。

## 适用范围

适用于：

- 页面样式
- 组件样式
- 局部布局样式
- Ant Design 等组件库配套的局部覆盖
- CSS Modules / Less / CSS 文件组织与使用方式

不适用于：

- 品牌视觉规范本身
- 全局主题设计决策
- 图标资源生成规则

## 基本原则

1. 优先复用项目现有样式体系
2. 优先使用组件库能力
3. 不写大段 inline style
4. 不污染全局样式
5. 不随意修改主题变量
6. 不为局部需求修改全局 CSS

## 样式优先级

新增样式时，按以下顺序判断：

1. 能否直接用组件库现有属性解决
2. 能否复用项目已有 class 或局部样式文件
3. 能否在当前页面或组件目录内新增局部样式
4. 只有明确跨页面稳定复用时，才考虑更高层级的公共样式能力

一句话原则：

- 能不用覆盖就不用覆盖
- 能局部解决就不升级到全局
- 能用组件能力解决，就不要自己重画一层

## 推荐做法

### 优先使用组件库能力

当 Ant Design 或项目组件库已能满足需求时，优先使用：

- 组件属性
- 尺寸能力
- `className`
- 合理布局组合
- 受控状态带来的视觉变化

不要为了一个颜色、一个间距或一个局部问题先去动全局主题或新建全局 CSS。

### 样式就近放置

推荐：

- 页面样式放在页面目录，如 `src/pages/<PageName>/index.less`
- 组件样式放在组件目录，如 `src/components/<ComponentName>/<ComponentName>.less`

不推荐：

- 为单页需求修改全局布局样式
- 为单个组件需求新建一份全局样式文件

### 样式职责和代码职责一致

- 页面样式负责页面布局、页面分区、页面私有外观
- 组件样式负责组件自身结构与视觉
- 全局样式只负责全局级内容，如基础字体、reset、外部库样式引入

不要把局部问题扔到全局样式里修。

## 作用域规则

### 默认局部作用域思维

即使使用 Less，也默认按“局部样式优先”的方式编写：

- 优先使用模块化 class
- 优先从当前组件根节点向下约束
- 避免裸标签和大范围裸类名覆盖

推荐：

```less
.container {
  display: flex;
}

.toolbar {
  margin-bottom: 12px;
}
```

不推荐：

```less
div {
  margin: 0;
}

.ant-btn {
  border-radius: 2px;
}
```

### `:global` 只用于必要覆盖

仅在以下场景考虑 `:global`：

- 覆盖组件库生成的内部类名
- 适配第三方组件固定 DOM 结构
- 解决微前端或父应用样式干扰

使用要求：

- 作用域尽量挂在当前页面或组件根 class 下
- 覆盖范围尽量小
- 明确知道自己在覆盖哪一层

推荐：

```less
.pageContainer {
  :global(.ant-tabs) {
    margin-bottom: 0;
  }
}
```

不推荐：

```less
:global(.ant-tabs) {
  margin-bottom: 0;
}
```

### 全局样式入口保持克制

`src/app.tsx` 中引入的全局样式只应用于：

- 全局基础样式
- 第三方库样式
- 明确跨全应用生效的底层样式

不应用于修复某一个页面的 margin、某一个表格的 padding、某一个弹窗的布局。

## Inline Style 规则

### 不写大段 inline style

禁止新增以下倾向：

- JSX 中堆大量布局样式
- 同一文件到处散落 `style={{ ... }}`
- 把可复用样式复制到多个节点

不推荐：

```tsx
<div
  style={{
    display: 'flex',
    alignItems: 'center',
    justifyContent: 'space-between',
    padding: '16px 24px',
    background: '#fff',
    borderRadius: 4,
  }}
/>
```

推荐：

```tsx
<div className={styles.toolbar} />
```

### 允许少量 inline style 的场景

以下场景可接受少量 inline style：

- 动态计算出的尺寸、位置、显隐值
- 一次性且非常简单的样式
- 组件库明确建议通过 `style` 传入的小范围视觉属性

例如：

- `style={{ height: containerHeight }}`
- `style={{ minWidth: 0 }}`

但仍应满足：

- 样式量很小
- 明显依赖运行时数据
- 不在多个节点重复出现

## 组件库覆盖规则

### 优先局部覆盖，不直接全局改库样式

可以：

- 在当前页面根 class 下局部覆盖某个 Ant Design 子结构
- 在组件私有根节点下局部覆盖第三方组件细节

不要：

- 全局直接覆盖 `.ant-btn`
- 全局直接覆盖 `.ant-table`
- 全局直接覆盖 `.ant-form-item`

### 覆盖前先确认是否已有组件能力

在覆盖前先确认：

- 是否已有 prop 能解决
- 是否已有 size / variant / token / className 插槽能力
- 是否只是没看清 DOM 结构就直接硬改样式

### 覆盖必须靠近问题发生点

比如表格 padding 不合适，应优先在该页面样式文件局部覆盖，而不是去全局布局或全局 CSS 里修整站表格。

## 主题变量与 Token

### 不随意修改主题变量

主题变量、全局 token、全局色板都属于高影响范围配置。

不要因为以下原因直接改：

- 某个页面颜色不满意
- 某个局部间距不顺眼
- 某个卡片边框想更重一点

只有明确属于全站统一设计调整时，才应该修改主题变量。

### 局部需求优先局部解决

局部视觉差异优先通过：

- 页面私有样式
- 组件私有样式
- 合理的 `className` 扩展

而不是先动全局 token。

## `!important` 规则

### 默认禁用

`!important` 只能作为最后手段，不能当默认方案。

禁止：

- 一覆盖失败就加 `!important`
- 连续多个属性一起加 `!important`
- 不分析权重与作用域就粗暴覆盖

### 可谨慎使用的场景

- 第三方库样式权重不可控
- 父应用样式污染导致局部无法正常显示
- 浏览器原生或外部注入样式无法通过正常选择器稳定覆盖

### 使用要求

- 只覆盖必要属性
- 只放在最小作用域里
- 最好通过注释说明原因

示例：

```less
.tableSection {
  :global(.ant-table-cell) {
    padding: 8px !important; // 父应用样式污染，局部兜底
  }
}
```

## 禁止行为

- 私自覆盖 antd 全局样式
- 在页面中写大量重复样式
- 把 `!important` 当常规手段
- 修改全局 layout 样式来修单页问题
- 在页面中散落大段 inline style
- 为单个页面去改 `src/app.tsx` 引入的全局样式入口
- 用裸选择器大范围覆盖第三方组件样式

## 推荐检查清单

1. 能否先用组件库能力解决，而不是直接写覆盖样式
2. 样式是否优先就近放在页面或组件目录
3. 是否误改了全局样式来修局部问题
4. 是否写了大段 inline style
5. 是否使用了过大的 `:global` 覆盖范围
6. 是否把 `!important` 当默认方案
7. 是否出现了同页重复样式却未抽成局部 class

## 一句话原则

样式优先复用现有体系，优先使用组件库能力，优先局部解决问题，不为单页需求污染全局。
