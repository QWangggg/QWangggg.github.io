**Input**: 可选地指定一个 `change name`。如果没有明确给出，则直接基于用户当前需求、当前项目上下文与仓库事实源继续执行。

## 适用范围

本 Skill 用于指导 AI Coding Agent 在前端项目中执行以下任务：

- 前端需求实现
- 页面新增或页面改造
- 组件新增或组件改造
- 接口接入与联调
- 状态与交互修正
- 权限与可见性控制
- 测试补充与回归修复

如任务本质上是需求澄清、规格沉淀或技术设计，而不是前端实现，应优先交给对应的 requirement / design agent，而不是直接进入编码。

## 输入优先级

执行任务时，按以下优先级读取上下文：

1. 仓库根 `AGENTS.md`
2. `docs/changes/{change-name}/spec.md`
3. `docs/changes/{change-name}/design.md`
4. 目标项目中与当前任务直接相关的现有代码
5. `agent/frontend/references/` 下的前端规则文档
6. 必要时通过 `knowledge` 获取补充事实

如果 spec、design 或事实源不完整，但不影响业务正确性，可以基于最小假设继续；如果会影响业务正确性，必须暂停确认，不得猜测实现。

## 标准开发节奏

每次执行前端任务，默认按以下节奏推进：

1. **读取上下文**
   - 先读 spec / design / 相关代码（不读代码就无法准确分型）
   - 识别目标页面、组件、接口所在位置，记录现有模式
2. **任务分型**
   - 基于读取结果，判断当前任务类型：
     - 页面开发 → 读 `frontend-page-rules.md`
     - 组件开发 → 读组件相关代码，按就近复用原则
     - 接口接入 → 读 `frontend-request-rules.md`
     - 状态与交互修正 → 读相关页面代码
     - 权限控制 → 读权限相关模式
     - 测试补充 → 读现有测试方式
3. **输出实现前设计说明**（见下方模板）
   - 如果命中 `Stop and Ask` 条件，必须在此步暂停，不得进入编码
4. **按最小改动完成实现**
   - 优先就地修改
   - 优先局部状态
   - 优先复用现有页面、组件、API、样式模式
   - 不为了"更工程化"额外新增全局能力
5. **完成验证与交付说明**
   - 补测试或沿用仓库现有测试方式
   - 无法运行验证时，明确说明未验证部分与风险

## 实现前设计说明模板

开始编码前，必须覆盖以下内容：

| 字段 | 说明 |
|------|------|
| **需求理解** | 本次要解决什么问题，边界是什么 |
| **核心输入** | 数据来源、参数来源、外部依赖 |
| **核心输出** | 页面表现、交互结果、数据结果 |
| **状态变更** | 计划新增哪些状态，为什么不能复用现有 |
| **副作用** | 是否涉及请求、事件、存储、跨应用通信 |
| **改动文件** | 修改哪些文件，是否新增文件，新文件落在哪个目录 |
| **引用 References** | 本次任务需要遵守的规则文档 |
| **关键假设** | 因信息缺失而做出的最小假设（仅列假设本身） |
| **不确定项** | 需要人工确认才能保证业务正确性的点（若有，必须暂停） |

> **区分**："关键假设"是"我认为 X 是 Y，据此继续"；"不确定项"是"我不知道 X 是什么，继续会出错，请确认"。有不确定项时不得继续编码。

## Stop and Ask 条件

遇到以下情况，必须暂停并等待确认，不得猜测继续：

- 权限码未知且影响按钮/页面可见性
- 提交接口契约不清晰（字段、枚举值、必填规则）
- 不清楚成功/失败后的跳转或刷新策略
- 涉及跨应用通信（Qiankun 主子应用）的新协议
- spec/design 对业务流程存在明显矛盾或空缺

## references 使用方式

根据任务类型按需读取：

- 页面开发 → `frontend-page-rules.md`
- 文件落点、抽象升级、目录争议 → `frontend-project-structure.md`
- 接口接入、命名、参数、错误处理 → `frontend-request-rules.md`
- 样式实现、局部覆盖、全局污染风险 → `frontend-style-rules.md`

References 是实现约束，若与仓库根规则或项目现状冲突，以仓库根规则和项目事实为准。

## 交付前自检

1. 是否满足当前需求，没有偷偷扩大范围
2. 是否遵守了目录职责边界
3. 是否复用了项目既有模式
4. 是否引入了不必要的新状态、新抽象或新依赖
5. 是否处理了空值、异常值、外部输入风险
6. 是否符合 TypeScript strict 预期
7. 是否考虑了 Qiankun 场景下的边界与兜底
8. 是否补充了必要测试，或明确说明验证缺口

## 不可妥协红线（Guardrails）

- 有不确定项时不得进入编码
- 不猜测权限码、接口契约、跨应用协议
- 不绕过 `src/api` 直接在页面写请求
- 不为单次需求创建全局抽象
- 不修改 `src/app.tsx` 全局样式入口来修单页问题
